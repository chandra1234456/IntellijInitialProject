Q1 What is Object-Oriented Programming (OOP)? What are its four pillars? ▼

OOP is a programming paradigm based on the concept of objects — self-contained units that bundle data (properties) and behavior (methods). It organises software design around data rather than functions and logic.

Encapsulation

Bundling data and methods; hiding internal state via access modifiers.

Inheritance

Deriving new classes from existing ones, enabling code reuse.

Polymorphism

Same interface, many forms — method overriding & overloading.

Abstraction

Hiding complex implementation details behind simple interfaces.

Q2 What is the difference between a Class and an Object? ▼
ClassA blueprint/template that defines properties and behaviors. It doesn't occupy memory by itself.
ObjectAn instance of a class created at runtime. It occupies memory and holds actual data values.
Kotlin Example
// Class = Blueprint
class Car(
    val brand: String,
    val speed: Int
) {
    fun drive() = println("$brand is driving at $speed km/h")
}

// Objects = Instances
val car1 = Car("Toyota", 120)
val car2 = Car("BMW", 200)
car1.drive()  // Toyota is driving at 120 km/h
car2.drive()  // BMW is driving at 200 km/h
Q3 What is a Constructor? Explain primary and secondary constructors in Kotlin. ▼

A constructor initialises an object when it's created. Kotlin has primary constructors (in the class header) and secondary constructors (inside the class body using constructor keyword).

Kotlin Example
class Person(
    val name: String,       // Primary constructor
    val age: Int
) {
    init {
        println("Person created: $name, age $age")
    }

    // Secondary constructor
    constructor(name: String) : this(name, 0) {
        println("Age defaulted to 0")
    }
}

val p1 = Person("Alice", 25)
val p2 = Person("Bob")      // Uses secondary constructor
Q4 What is the difference between val and var in Kotlin? How does it relate to OOP? ▼
valImmutable reference — cannot be reassigned after initialisation (like Java's final). Encourages encapsulation.
varMutable reference — can be reassigned. Should be used with care to maintain encapsulation.

class User {

Q1 What is Encapsulation? How is it achieved in Kotlin? ▼

Encapsulation is the practice of bundling data and methods together and restricting direct access to internal state. In Kotlin, it's achieved using private properties with public getters/setters, or Kotlin's built-in property accessor syntax.

Kotlin Example
class Employee {
    private var _salary: Double = 0.0

    // Custom getter/setter
    var salary: Double
        get() = _salary
        set(value) {
            if (value >= 0) _salary = value
            else println("Invalid salary")
        }

    private var _name: String = ""
    val name: String get() = _name.uppercase()

    fun setName(name: String) {
        if (name.isNotBlank()) _name = name
    }
}

val emp = Employee()
emp.setName("alice")
emp.salary = 50000.0
println(emp.name)    // ALICE
println(emp.salary)  // 50000.0
emp.salary = -100.0 // Invalid salary
Q2 What is a data class in Kotlin and how does it support encapsulation? ▼

A data class automatically generates equals(), hashCode(), toString(), copy(), and componentN() functions. It promotes encapsulation by making all primary constructor properties val (immutable) by convention.

Kotlin Example
data class UserProfile(
    val id: Int,
    val name: String,
    val email: String
)

val user1 = UserProfile(1, "Alice", "alice@mail.com")
val user2 = user1.copy(name = "Bob")   // new object, original unchanged
println(user1)  // UserProfile(id=1, name=Alice, email=alice@mail.com)
println(user1 == user2)  // false

Q1 What is Inheritance? How does Kotlin handle it differently from Java? ▼

Inheritance lets a class acquire properties and methods from another class. In Kotlin, all classes are final by default — you must explicitly mark them open to allow inheritance. In Java, classes are open by default.

Kotlin Example
open class Animal(val name: String) {
    open fun speak() = println("$name makes a sound")
}

class Dog(name: String) : Animal(name) {
    override fun speak() = println("$name says: Woof!")
}

class Cat(name: String) : Animal(name) {
    override fun speak() = println("$name says: Meow!")
}

val animals: List<Animal> = listOf(Dog("Rex"), Cat("Whiskers"))
animals.forEach { it.speak() }
// Rex says: Woof!
// Whiskers says: Meow!

Q1 What is Polymorphism? Explain compile-time vs runtime polymorphism. ▼

Polymorphism means "many forms" — the ability to process objects differently based on their type.

Compile-timeMethod overloading — same method name, different parameters. Resolved at compile time.
RuntimeMethod overriding — subclass provides specific implementation. Resolved at runtime via dynamic dispatch.
Overloading (Compile-time)
class Calculator {
    fun add(a: Int, b: Int) = a + b
    fun add(a: Double, b: Double) = a + b
    fun add(a: Int, b: Int, c: Int) = a + b + c
}
val calc = Calculator()
println(calc.add(1, 2))          // 3
println(calc.add(1.5, 2.5))     // 4.0
println(calc.add(1, 2, 3))      // 6
Overriding (Runtime)
open class Shape {
    open fun area(): Double = 0.0
}
class Circle(val r: Double) : Shape() {
    override fun area() = Math.PI * r * r
}
class Rectangle(val w: Double, val h: Double) : Shape() {
    override fun area() = w * h
}
val shapes: List<Shape> = listOf(Circle(5.0), Rectangle(4.0, 6.0))
shapes.forEach { println("Area: ${"%.2f".format(it.area())}") }
OutputArea: 78.54
Area: 24.00

Q1 What is Abstraction and how is it used in Android Architecture? ▼

Abstraction hides complexity by exposing only what's necessary. In Android's MVVM architecture, the Repository pattern is a great example — the ViewModel doesn't care if data comes from a network or database.

Android Repository Pattern
// Abstract interface — hides implementation
interface UserRepository {
    suspend fun getUser(id: Int): User
    suspend fun saveUser(user: User)
}

// Concrete implementation (hidden from ViewModel)
class UserRepositoryImpl(
    private val api: ApiService,
    private val db: UserDao
) : UserRepository {
    override suspend fun getUser(id: Int): User {
        return db.getUser(id) ?: api.fetchUser(id).also { db.insert(it) }
    }
    override suspend fun saveUser(user: User) = db.insert(user)
}

// ViewModel only knows the interface
class UserViewModel(private val repo: UserRepository) : ViewModel() {
    fun loadUser(id: Int) { /* repo.getUser(id) */ }
}

Easy Problem 1: Implement a Stack using OOP

Design a generic Stack class with push, pop, peek, and isEmpty operations using encapsulation principles.

Solution
Kotlin
class Stack<T> {
    private val items = mutableListOf<T>()

    fun push(item: T) = items.add(item)

    fun pop(): T {
        if (isEmpty()) throw EmptyStackException()
        return items.removeAt(items.size - 1)
    }

    fun peek(): T {
        if (isEmpty()) throw EmptyStackException()
        return items.last()
    }

    fun isEmpty() = items.isEmpty()
    fun size() = items.size

    override fun toString() = items.toString()
}

// Usage
val stack = Stack<Int>()
stack.push(10)
stack.push(20)
stack.push(30)
println(stack.peek())  // 30
println(stack.pop())   // 30
println(stack)         // [10, 20]
Output30

Medium Problem 3: Shape Hierarchy with Abstract Class

Build a shape hierarchy using abstract class and polymorphism to calculate area and perimeter.

Kotlin
abstract class Shape(val color: String) {
    abstract fun area(): Double
    abstract fun perimeter(): Double
    open fun describe() =
        println("$color ${this::class.simpleName}: area=${"%.2f".format(area())}, perimeter=${"%.2f".format(perimeter())}")
}

class Circle(val radius: Double, color: String) : Shape(color) {
    override fun area() = Math.PI * radius * radius
    override fun perimeter() = 2 * Math.PI * radius
}

class Rectangle(val w: Double, val h: Double, color: String) : Shape(color) {
    override fun area() = w * h
    override fun perimeter() = 2 * (w + h)
}

class Triangle(val a: Double, val b: Double, val c: Double, color: String) : Shape(color) {
    override fun area(): Double {
        val s = (a + b + c) / 2
        return Math.sqrt(s * (s-a) * (s-b) * (s-c))
    }
    override fun perimeter() = a + b + c
}

val shapes = listOf(
    Circle(5.0, "Red"),
    Rectangle(4.0, 6.0, "Blue"),
    Triangle(3.0, 4.0, 5.0, "Green")
)
shapes.forEach { it.describe() }
println("Total area: ${"%.2f".format(shapes.sumOf { it.area() })}")
OutputRed Circle: area=78.54, perimeter=31.42
Blue Rectangle: area=24.00, perimeter=20.00
Green Triangle: area=6.00, perimeter=12.00
Total area: 108.54
Medium Problem 4: Builder Pattern for Android AlertDialog

Implement the Builder design pattern (commonly used in Android's AlertDialog, Retrofit, OkHttp) to construct complex objects step by step.

Kotlin
data class AlertConfig(
    val title: String,
    val message: String,
    val positiveButton: String,
    val negativeButton: String?,
    val isCancelable: Boolean
) {
    class Builder {
        private var title = ""
        private var message = ""
        private var positiveButton = "OK"
        private var negativeButton: String? = null
        private var isCancelable = true

        fun title(t: String) = apply { title = t }
        fun message(m: String) = apply { message = m }
        fun positiveButton(b: String) = apply { positiveButton = b }
        fun negativeButton(b: String) = apply { negativeButton = b }
        fun cancelable(c: Boolean) = apply { isCancelable = c }
        fun build() = AlertConfig(title, message, positiveButton, negativeButton, isCancelable)
    }
}

val config = AlertConfig.Builder()
    .title("Delete Item")
    .message("Are you sure you want to delete?")
    .positiveButton("Delete")
    .negativeButton("Cancel")
    .cancelable(false)
    .build()

println(config)

Medium Problem 5: Observer Pattern (LiveData-style)

Implement the Observer pattern — the foundation of Android's LiveData and reactive UI updates.

Kotlin
// Observer interface
interface Observer<T> {
    fun onChanged(value: T)
}

// Observable (simplified LiveData)
class MutableLiveData<T> {
    private val observers = mutableListOf<Observer<T>>()
    private var _value: T? = null

    var value: T?
        get() = _value
        set(newVal) {
            _value = newVal
            notifyObservers()
        }

    fun observe(observer: Observer<T>) = observers.add(observer)

    private fun notifyObservers() {
        _value?.let { v -> observers.forEach { it.onChanged(v) } }
    }
}

// Usage
val usernameLiveData = MutableLiveData<String>()

usernameLiveData.observe(object : Observer<String> {
    override fun onChanged(value: String) = println("UI updated: $value")
})

usernameLiveData.value = "Alice"
usernameLiveData.value = "Bob"
OutputUI updated: Alice
UI updated: Bob
Hard Problem 6: Strategy Pattern for Sorting

Implement the Strategy design pattern to switch sorting algorithms at runtime — demonstrating polymorphism and abstraction.

Kotlin
// Strategy interface
interface SortStrategy {
    fun sort(list: MutableList<Int>): List<Int>
}

class BubbleSort : SortStrategy {
    override fun sort(list: MutableList<Int>): List<Int> {
        for (i in list.indices) {
            for (j in 0 until list.size - i - 1) {
                if (list[j] > list[j + 1]) {
                    val tmp = list[j]; list[j] = list[j+1]; list[j+1] = tmp
                }
            }
        }
        return list
    }
}

class QuickSort : SortStrategy {
    override fun sort(list: MutableList<Int>): List<Int> {
        if (list.size <= 1) return list
        val pivot = list[list.size / 2]
        val left = list.filter { it < pivot }.toMutableList()
        val mid = list.filter { it == pivot }
        val right = list.filter { it > pivot }.toMutableList()
        return sort(left) + mid + sort(right)
    }
}

// Context class
class Sorter(var strategy: SortStrategy) {
    fun sort(list: MutableList<Int>) = strategy.sort(list)
}

val data = mutableListOf(64, 34, 25, 12, 22, 11, 90)
val sorter = Sorter(BubbleSort())
println(sorter.sort(data.toMutableList()))  // Bubble
sorter.strategy = QuickSort()
println(sorter.sort(data.toMutableList()))  // Quick
Output[11, 12, 22, 25, 34, 64, 90]
[11, 12, 22, 25, 34, 64, 90]
Hard Problem 7: Generic Repository with Sealed State

Build a type-safe generic repository that wraps results in a sealed class — a real-world Android MVVM pattern.

Kotlin
// Sealed class for UI state
sealed class UiState<out T> {
    object Idle : UiState<Nothing>()
    object Loading : UiState<Nothing>()
    data class Success<T>(val data: T) : UiState<T>()
    data class Error(val message: String) : UiState<Nothing>()
}

// Generic base repository
abstract class BaseRepository {
    protected suspend fun <T> safeApiCall(
        apiCall: suspend () -> T
    ): UiState<T> {
        return try {
            UiState.Success(apiCall())
        } catch (e: Exception) {
            UiState.Error(e.message ?: "Unknown error")
        }
    }
}

data class User(val id: Int, val name: String)

class UserRepository : BaseRepository() {
    suspend fun getUser(id: Int): UiState<User> = safeApiCall {
        if (id > 0) User(id, "User_$id")
        else throw IllegalArgumentException("Invalid ID")
    }
}

// In ViewModel / UI
suspend fun main() {
    val repo = UserRepository()
    when (val state = repo.getUser(5)) {
        is UiState.Success -> println("Got user: ${state.data.name}")
        is UiState.Error   -> println("Error: ${state.message}")
        else -> {}
    }
}
OutputGot user: User_5
