Q1: Tightly Coupled?
  • Two classes are tightly coupled when one depends directly on the internal implementation of another.
  • If one class changes, the other class also needs modification.
  • This reduces flexibility and increases maintenance cost.

class MainActivity {
    UserRepository repo = new UserRepository();

    void loadUser() {
        repo.getUser();
    }
}
Here, MainActivity directly creates UserRepository → strong dependency.
Problems:
  • Hard to test (no mocking possible)
  • Hard to maintain
  • Low reusability
Q2: Loosely Coupled?
  • Components interact through interfaces instead of concrete classes.
  • Changes in one module do not heavily affect others.
  • Improves scalability and testability.

interface UserRepo {
    void getUser();
}

class UserRepository implements UserRepo {
    public void getUser() {
        // implementation
    }
}

class MainActivity {
    UserRepo repo;

    MainActivity(UserRepo repo) {
        this.repo = repo;
    }
}
Now MainActivity depends on abstraction (interface), not concrete class.
Benefits:
  • Easier testing (mocking possible)
  • Better maintainability
  • Flexible architecture
Android Context

Loose coupling is commonly used in Android using:

  • MVVM architecture
  • Dependency Injection (Hilt / Dagger)
  • Repository pattern + ViewModel separation
Simple Analogy

Tight coupling: A TV remote that works only with one specific TV brand

Loose coupling: A universal remote that works with many TVs
