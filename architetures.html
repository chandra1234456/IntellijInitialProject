Android Architecture Patterns – Diagrams & Interview Notes

1️⃣ MVVM (Model–View–ViewModel)

Graphical Representation

Package Structure

ui/ └── main/ ├── MainActivity ├── MainViewModel data/ ├── repository/ ├── model/ └── datasource/

Lifecycle

  • ViewModel survives configuration changes
  • Destroyed when Activity/Fragment is finished

Interview Notes

  • Most recommended by Google
  • ViewModel never holds View reference
  • Works with LiveData / StateFlow

2️⃣ MVI (Model–View–Intent)

Graphical Representation

Package Structure

ui/ └── screen/ ├── ScreenView ├── ScreenIntent ├── ScreenState └── ScreenViewModel

Lifecycle

  • Unidirectional data flow
  • Single immutable state

Interview Notes

  • Very popular with Jetpack Compose
  • Easier debugging
  • More boilerplate than MVVM

3️⃣ Clean Architecture

Graphical Representation

Package Structure

presentation/ ├── ui/ └── viewmodel/ domain/ ├── usecase/ └── model/ data/ ├── repository/ ├── datasource/ └── mapper/

Lifecycle

  • Domain layer independent of Android
  • Presentation depends on Domain

Interview Notes

  • Highly testable
  • Scalable for large apps
  • More complex structure

4️⃣ MVC (Model–View–Controller)

Graphical Representation

Package Structure

ui/ ├── Activity (Controller + View) model/ └── Data classes

Lifecycle

  • Controller tied to Activity lifecycle
  • No separation of concerns

Interview Notes

  • Not recommended for Android
  • Leads to Massive Activities
  • Hard to test

5️⃣ Quick Interview Comparison

  • Best for Android: MVVM + Clean Architecture
  • Best for Compose: MVI
  • Worst: MVC
  • Most scalable: Clean Architecture
